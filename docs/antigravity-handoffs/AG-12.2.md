## 변경 파일과 각 변경 목적

- `src/ledger/review.ts` (신규/수정):
  - 기존 서버 측 계산 로직에 의존하던 검증을 프론트엔드에서도 수행할 수 있도록, 순수 함수인 `validateRow`를 작성했습니다. 오류 원인을 필드 단위로 매핑하여 반환합니다.
  - 문자열 좌우 공백 제거, 연속 공백 축소, 소문자 변환을 수행하는 `normalizeItemName`을 작성했습니다.
  - 기저장된 레코드 및 입력 중인 타 행과 비교하여 중복을 찾는 `findDuplicateCandidates` 순수 함수를 분리했습니다. 이번 수정에서 **0원 총액 제외 규칙을 제거하여 모든 총액에 동일하게 완전 일치 기준이 적용**되도록 했습니다.
  - 수기 세액 모드일 때는 `taxAmount`에만, 자동 세액 모드일 때는 `taxRate`에만 오류를 매핑하도록 분리했습니다.
- `src/ledger/review.test.ts` (신규/수정):
  - 새로 작성한 비즈니스 로직(오류 필드 → 셀 매핑, 동일 조건만 중복 탐지 등)에 대한 테스트입니다.
  - 수기/자동 세액 입력 모드별로 오류 필드가 정확하게 분리되는지 확인하는 테스트를 추가했습니다.
  - 총액이 0원인 경우라도 나머지 기준이 완전 일치하면 중복 경고 대상이 되는지 확인하는 테스트를 추가했습니다.
  - 같은 세션에서 저장 직후 데이터가 갱신되었을 때 중복이 탐지되는지 확인하는 테스트를 추가했습니다.
- `src/app/ledger/page.tsx`:
  - 렌더링 시마다 각 행을 `validateRow`와 `findDuplicateCandidates`로 검사하고, 오류가 발생한 `<input>`에는 `aria-invalid="true"`를 부여합니다.
  - 행의 우측 끝에 "상태" 열(`<td>`)을 추가하여 텍스트 형태의 오류 설명(행 요약)과 중복 의심 경고를 노출합니다.
  - 저장 API 호출이 성공한 경우 응답값(`records`)을 현재 `candidatesSource`에 병합하여, **같은 세션에서 동일 내역을 재입력할 때 즉시 중복 경고가 노출되도록 최신화**했습니다.
  - 하단 도움말에 후보 탐지가 최근 저장된 200건 내에서만 이뤄진다는 사실을 안내했습니다.
- `src/app/ledger/page.module.css`:
  - `aria-invalid="true"`가 부여된 `<input>`에 붉은색 테두리와 배경색을 적용하여 시각적으로 오류 셀임을 명확히 표시합니다.
  - 오류 요약(`errorSummary`) 및 경고 요약(`warningSummary`)의 스타일을 지정했습니다.

- 브라우저 상호작용 및 DOM 렌더링에 대한 E2E (Playwright/Cypress 등) 테스트는 실행하지 않았습니다.
- 키보드 Enter 이동, 엑셀 셀 복사/붙여넣기에 대한 브라우저 이벤트 직접 발송 테스트는 자동화 환경 한계로 생략되었습니다.

## 중복 후보 기준과 업무상 한계

- **후보 기준**: 동일 회사, 동일 현장, 동일 비용분류, 동일 발생일, **정규화된 품명**, 동일 계산 총액(전액)이 완전히 일치하는 경우. (금액이 0.00이라도 다른 모든 조건이 일치하면 후보로 탐지됨)
- **정규화 규칙**: 좌우 공백 제거, 다중 공백 단일화, 영문 대소문자 무시.
- **업무상 한계 및 오탐/누락 가능성**: 품명에 의미 없는 특수문자나 조사가 추가되었거나 띄어쓰기가 비정상적으로 들어간 경우 완벽히 거르지 못하여 실제로는 동일 건이어도 중복으로 인식하지 못할 수 있습니다(누락). 반대로 의도적으로 같은 날 같은 현장에 같은 금액으로 두 번 청구된 정상적인 거래일 수 있으나 경고가 발생할 수 있습니다(오탐). 또한, 서버의 GET API(`200건` 제한) 내에서만 비교하므로 과거 내역 전체를 포괄하지 못합니다. 이는 어디까지나 **기술적 주의 알림**일 뿐이며 저장을 절대 차단하지 않습니다.

## 확인되지 않은 업무 규칙

- 중복 탐지를 "기술적 알림" 이상의 업무적 확정으로 승격해야 할지 여부.
- 중복이 탐지되었을 때 사용자가 "저장 사유(Reason)"를 남겨야 하는지 여부(현재는 강제하지 않음).
- 에러가 발생한 행(오류 셀 존재)이 배열에 포함되어 있을 때 무시하고 유효한 행만 먼저 저장할지, 아니면 전체 저장을 막을지 여부. 현재는 모든 행을 화면에 남겨두고 유효한 행(품명/일자 존재)들에 대해서 계산 시 에러가 있으면 전체 배열 저장을 중지시킵니다(저장 API의 기존 로직).
